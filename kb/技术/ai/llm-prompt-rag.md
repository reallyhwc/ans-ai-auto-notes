# Prompt 与 RAG 体系

> 最后整理: 2026-05-04 | 来源: 多轮对话

## 一句话定位

Prompt 是怎么跟 LLM 说话，RAG 是给 LLM 外挂知识库。两者结合，让 LLM 从"只会背书的书呆子"变成"能查参考资料的助手"。

> 关联: [llm](./llm.md) — LLM 核心原理 | [llm-agent-mcp](./llm-agent-mcp.md) — Agent 与 MCP 生态

---

## 1. Prompt：怎么跟大模型说话

Prompt 就是你输入给模型的文本。同一个模型，不同 Prompt，效果天差地别。

### 1.1 Zero-shot：直接问

```
"告诉我怎么做红烧肉"  → 可以得到答案，但可能很泛
```

### 1.2 Few-shot：给几个例子再问

```
"英语→中文:
 hello → 你好
 world → 世界
 cat → 猫
 dog →"                    → 模型从例子理解了"你要翻译"
```

模型不需要重新训练，从 Prompt 里的几个例子就学会了任务模式——这叫 **In-context Learning（上下文学习）**。

### 1.3 Chain-of-Thought（思维链）：让它一步步想

```
❌ "一个班有 30 个学生，男生比女生多 4 个，问男生有多少人？"
   → 模型直接给答案，容易错

✅ "一个班有 30 个学生，男生比女生多 4 个，问男生有多少人？请一步步推理。"
   → 模型:
     "设女生人数为 x，则男生为 x+4
      总人数: x + (x+4) = 30
      2x + 4 = 30, 2x = 26, x = 13
      男生 = 13 + 4 = 17 人"
   → 准确率大幅提升
```

**一句话：让模型把中间推理过程说出来，它能算得更准。** 这是 Prompt 工程里最重要的技巧之一。

## 2. RAG（检索增强生成）：给模型外挂知识库

### 2.1 问题

LLM 的知识截止于训练数据。问它"今天天气怎样"或"公司内部 API 文档里的那个字段叫什么"——它不知道，或者更糟，它**会编一个看起来很合理的错误答案**（幻觉）。

### 2.2 方案

```
传统方式:
  用户提问 → LLM → 回答 (只能靠训练记忆)

RAG 方式:
  用户提问 → 先检索外部知识库 → 把检索结果一起塞给 LLM → 回答
            ↑
    ┌──────────────────┐
    │  外部知识库       │
    │  (文档/数据库/API) │
    └──────────────────┘
```

### 2.3 具体例子

```
用户: "公司内部支付接口的退款 API 是哪个？"

Step 1 — 检索:
  从公司文档库中搜索"退款 API"
  → 找到: "退款接口：POST /api/v2/payment/refund, 参数: orderId, amount, reason"

Step 2 — 增强:
  "根据以下文档回答问题:
   【文档内容】退款接口：POST /api/v2/payment/refund, 参数: orderId, amount, reason
   【问题】公司内部支付接口的退款 API 是哪个？"

Step 3 — 生成:
  LLM → "退款 API 是 POST /api/v2/payment/refund，需要传 orderId、amount、reason 三个参数。"
```

**RAG 的核心价值：让模型能回答训练数据之外的问题，同时大幅减少幻觉。**

## 3. RAG 检索细节与 Prompt 拼接实例

### 3.1 检索流程

```
用户: "我买的衣服不合身，怎么退？"
            │
            ▼
┌──────────────────────────┐
│  Step 1: 用户问题         │
│  Embedding 模型 → 向量    │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  Step 2: Milvus 相似搜索  │
│  使用 ANN 算法（HNSW/IVF） │  ← 不是逐个比对，类似 B+ 树索引
│  毫秒级找到 top-3         │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  Step 3: 取出文档原文      │
│  top-1: "退款步骤..."     │
│  top-2: "退货流程..."     │
│  top-3: "退换政策..."     │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  Step 4: LangChain 拼接   │  ← 检索由向量数据库 SDK 完成，
│  Prompt 发给 LLM          │    LangChain 负责"从检索结果到 Prompt"
└──────────────────────────┘
```

### 3.2 拼接后发给 LLM 的完整 Prompt

```
你是本公司电商客服助手"小易"，请严格根据以下参考资料回答用户问题。
如果参考资料中没有相关信息，请回复"抱歉，我暂时无法回答这个问题，
建议您联系人工客服"。

【参考资料】
---
来源1 (相似度: 97%):
Q: 怎么退款？
A: 登录APP → 点击"我的订单" → 找到对应订单 → 点击"申请退款" → 
   选择退款原因 → 提交申请 → 等待审核（1-3个工作日）→ 退款到原支付方式

来源2 (相似度: 91%):
退货流程：用户在APP提交退货申请 → 系统审核 → 审核通过后用户寄回商品 →
仓库确认收货 → 发起退款。全程约需5-7个工作日。

来源3 (相似度: 85%):
退换货政策：支持7天无理由退换货（商品未经使用、不影响二次销售）。
非质量问题退货运费由买家承担，质量问题由平台承担。
---

【对话历史】
用户: 你好
客服: 您好！有什么可以帮助您的？

【用户当前问题】
我买的衣服不合身，怎么退？
```

LLM 不是凭"记忆"回答——而是**阅读参考资料，理解后组织回答**。

## 4. Embedding 模型详解

### 4.1 它和大模型是两码事

```
LLM（大模型）:
  输入: 文本 → 输出: 文本
  用途: 生成回答、推理
  例子: GPT-4, DeepSeek-V3, Qwen-Max
  大小: 几百 GB，速度慢

Embedding 模型:
  输入: 文本 → 输出: 向量 [0.23, -0.15, 0.78, ...]
  用途: 判断文本相似度
  例子: text-embedding-3-small (OpenAI), bge-large (BAAI), m3e (Moka AI)
  大小: 几百 MB，速度快
```

### 4.2 训练原理：对比学习

```
训练目标: 语义相似的文本 → 向量距离近

给模型一对文本:
  "怎么退款" 和 "如何退货" → 正样本(语义相似) → 拉近向量距离
  "怎么退款" 和 "今天天气" → 负样本(语义无关) → 推远向量距离

几百万次"拉近/推远"后:
  → Embedding 模型学会了把语义相似的文本映射到相近的向量
```

### 4.3 RAG 中两个模型的协作

```
离线:
  文档 → [Embedding 模型] → 向量 → Milvus

在线:
  用户问题 → [Embedding 模型] → 向量 → Milvus 检索 → 相关文档
              ↓
  拼接 Prompt → [LLM] → 回答
```

**Embedding 模型负责"找"，LLM 负责"说"。** 两个独立的模型，各司其职。

> 关联: [transformer](./transformer.md) — Embedding 模型通常也基于 Transformer（Encoder-only，如 BERT 架构）

## 5. 向量数据库 & Milvus：给 LLM 装"搜索引擎"

### 5.1 为什么需要向量数据库

LLM 的知识是训练时冻结的。要让 LLM 回答训练数据之外的内容，需要把外部知识变成它能搜索的格式——这就是向量数据库干的活。

### 5.2 核心原理：把文字变成空间坐标

```
"退款接口" → Embedding 模型 → [0.23, -0.15, 0.78, ..., 0.41]  (768维向量)
"支付API"  → Embedding 模型 → [0.21, -0.13, 0.75, ..., 0.39]

余弦相似度 = 0.97 → 这两个文本高度相关！

"请假流程" → Embedding 模型 → [-0.45, 0.32, -0.11, ..., 0.67]
余弦相似度("退款接口", "请假流程") = 0.12 → 不相关
```

### 5.3 Milvus 是什么

开源向量数据库，Zilliz 出品，专为 AI 场景设计：

| 特性 | 说明 |
|------|------|
| **十亿级向量** | 支持超大规模索引 |
| **毫秒级检索** | 近似最近邻（ANN）算法 |
| **混合查询** | 向量相似度 + 标量过滤（如"只查 2024 年的文档"） |
| **类比** | MySQL 存行 → Milvus 存向量；Redis 存键值 → Milvus 存向量 |

其他主流向量数据库：Pinecone（云托管）、Weaviate、Qdrant、Chroma（轻量）。

## 6. LangChain + 向量数据库 + 大模型：三者的真正关系

### 6.1 两个"喂"的区别（核心澄清）

```
训练大模型（预训练阶段）:
  喂的是 → 海量互联网文本（几万亿 token）
  目的   → 让模型学会语言模式、推理、知识
  成本   → 几千万美元，几个月，几千张 GPU

向量数据库（RAG 阶段）:
  喂的是 → 你的私有文档（客服历史、产品手册、公司制度）
  目的   → 让模型能够引用这些文档回答你的问题
  成本   → 几乎免费
```

**向量数据库不是用来训练模型的——是用来查资料的。** 类比：向量数据库是模型的"参考书"，不是模型的"教材"。

### 6.2 客服机器人完整技术栈

```
┌─────────────────────────────────────────────────────────┐
│  离线准备（只做一次，数据更新时重做）                       │
│                                                         │
│  历史客服对话 (Q&A 对)                                    │
│    → 清洗 + 切片                                         │
│    → Embedding 模型 → 每段变成 768 维向量                  │
│    → 存入 Milvus                                         │
│                                                         │
│   Milvus 中:                                             │
│    Q1: "怎么退款"  → [0.23, -0.15, 0.78, ...]            │
│    A1: "登录APP→我的订单→申请退款" → [0.21, -0.13, ...]    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  在线服务（用户每次提问时）                                 │
│                                                         │
│  用户: "我买的衣服不合身，怎么退？"                         │
│    → Embedding → [0.22, -0.14, 0.76, ...]                │
│    → Milvus 搜索 → 最相关 3 条文档                        │
│    → LangChain 把文档拼入 Prompt                          │
│    → 大模型 (DeepSeek/Qwen/GPT) 生成最终回答               │
└─────────────────────────────────────────────────────────┘
```

### 6.3 常见疑问

| 疑问 | 答案 |
|------|------|
| LangChain 底层封装好调用大模型？ | **是的**，但不止——它还串联了检索、记忆、工具调用 |
| 向量数据库输出什么，依赖我们喂什么？ | **是的**，喂的是私有文档（不是训练数据），检索出来拼入 Prompt |
| 客服历史数据喂给向量数据库还是大模型？ | **喂给向量数据库**做索引，大模型只负责"根据检索到的参考资料回答" |
| 需要自己买 GPU 吗？ | **不需要**，调 API 按量付费。DeepSeek 极便宜（¥1/百万 token） |

## 7. LangChain：LLM 的编排框架

**LangChain 不是模型，是一个编排框架。** 类比：Spring Boot 不处理 HTTP 请求（是 Tomcat 干的），但它把控制器、服务、数据库串起来。

### 7.1 没有框架 vs 有框架

```python
# 不用框架：几百行胶水代码自己拼

# 用 LangChain:
from langchain.chains import RetrievalQA
qa = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(),
    retriever=milvus.as_retriever()
)
qa.run("退款 API 是哪个？")  # 就几行
```

### 7.2 LangChain 核心做了四件事

| 功能 | 做什么 | 类比 Java |
|------|--------|-----------|
| **Prompt 模板** | 变量注入、Few-shot 示例管理 | JSP/Thymeleaf 模板引擎 |
| **对话记忆** | 自动管理多轮对话历史，处理上下文窗口溢出 | HTTP Session 管理 |
| **Chain（链）** | 把 LLM + 工具 + 检索 串成流水线 | Spring Integration / Pipeline |
| **Agent 封装** | 思考→行动→观察循环，自动决定调用哪个工具 | 工作流引擎 |

### 7.3 主流 LLM 框架

| 框架 | 语言 | 特点 |
|------|------|------|
| **LangChain** | Python/JS | 功能最全，生态最大 |
| **LlamaIndex** | Python | 专注数据索引和 RAG |
| **Semantic Kernel** | C#/Python/Java | 微软出品，企业级 |
| **Spring AI** | Java | Spring 生态，Java 程序员最熟悉 |
