# AI 时代程序员进阶：从 Vibe Coding 到 Harness Engineering

> 最后整理: 2026-05-25 | 来源: 对话讨论

> 关联: [agent-development-practice](../应用/agent-development-practice.md)

## 三阶段全景图

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   Vibe Coding          Spec Coding           Harness Eng.       │
│   ─────────────        ─────────────         ─────────────      │
│   "嘿 AI，帮我写个     "按这个规格实现，      "我设计系统让 AI     │
│    登录页面"            测试必须通过"          持续稳定地产出"      │
│                                                                 │
│   ← 低杠杆 ─────────────────────────────────── 高杠杆 →         │
│   ← 低可靠性 ────────────────────────────────── 高可靠性 →       │
│   ← 随机产出 ────────────────────────────────── 确定性产出 →     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

## 第一阶段：Vibe Coding（氛围编程）

**核心特征**：用自然语言模糊描述需求，让 AI 生成代码，凭"感觉"判断结果对不对。

```
典型工作方式:
  用户: "帮我写一个用户注册接口"
  AI:   生成了一堆代码
  用户: "看起来差不多，能跑就行" ← 这就是 vibe

问题:
  - 不确定生成的代码是否覆盖了边界情况
  - 不确定是否符合项目规范
  - 出了 bug 就再问一次 AI，"帮我修一下"
  - 对 AI 产出没有验证手段
```

**实际工作表现**：
- 把需求直接丢给 AI，拿到代码粘贴进项目
- 遇到报错截图再丢给 AI，循环往复
- 不写测试，不做 review，"能跑就行"
- 产出质量完全取决于 prompt 写得好不好 + AI 的"心情"

**适用场景**：个人小项目、原型验证、一次性脚本。

---

## 第二阶段：Spec Coding（规格编程）

**核心特征**：先写清楚"要什么"（规格/测试），再让 AI 按规格实现，用自动化手段验证产出。

```
典型工作方式:
  1. 先写规格:
     - 邮箱格式校验
     - 密码至少 8 位，含大小写
     - 重复注册返回 409
     - 成功返回 201 + token

  2. 先写测试（TDD）:
     test_register_success()
     test_register_duplicate_email()
     test_register_invalid_password()

  3. 让 AI 实现: "让所有测试通过"

  4. 验证: 跑测试 → 全绿 → 确认完成
     ← 不再靠"感觉"，而是靠测试结果
```

**与 Vibe Coding 的关键区别**：

| 维度 | Vibe Coding | Spec Coding |
|------|-------------|-------------|
| **需求表达** | 自然语言模糊描述 | 结构化规格 + 测试用例 |
| **验证方式** | 人眼看"差不多" | 自动化测试通过 |
| **迭代方式** | "帮我改一下" | "让这个测试通过" |
| **产出确定性** | 随机，靠运气 | 确定，测试保证 |
| **错误发现** | 上线后才知道 | 写完就知道 |

**实际工作表现**：
- 拿到需求先写设计文档/接口规格
- 先写测试用例，定义"什么叫做对"
- 给 AI 明确的约束条件和验收标准
- AI 的产出必须通过 CI/CD 验证

---

## 第三阶段：Harness Engineering（驾驭工程）

**核心特征**：不只是让 AI 写一段代码，而是**设计一整套系统和流程，让 AI 能持续、可靠、大规模地产出高质量工作**。

```
不是: "AI，帮我实现这个功能"
而是: "我来设计一套机制，让 AI 能自主完成以下流程:
      1. 理解需求（通过 Skill 引导）
      2. 拆解任务（通过 Plan 模板）
      3. 逐步实现（通过 TDD Skill 约束）
      4. 自我验证（通过测试 + lint）
      5. 自我审查（通过 review Skill）
      6. 持续改进（通过 memory 反馈循环）"
```

**Harness Engineer 做的事情**：

```
┌─────────────────────────────────────────────┐
│         Harness Engineer 的工作              │
│                                             │
│  设计 Skill      → 把专家经验编码成 SOP      │
│  设计 Memory     → 让 AI 记住项目偏好和教训   │
│  设计 Guard Rail → 设置安全边界防止 AI 越界   │
│  设计 Feedback   → 从错误中自动学习改进       │
│  设计 Workflow   → 编排多 Agent 协作流程      │
│  设计验证机制     → 确保 AI 产出可靠           │
│                                             │
│  类比: 不是骑马，是设计赛道 + 训练规则         │
└─────────────────────────────────────────────┘
```

**与 Spec Coding 的关键区别**：

| 维度 | Spec Coding | Harness Engineering |
|------|-------------|---------------------|
| **关注点** | 单次任务的输入输出 | 系统级的持续产出能力 |
| **工作对象** | 一段代码、一个功能 | 整套工作流、Skill、Memory |
| **时间维度** | 当下这一次做对 | 以后每一次都做对 |
| **杠杆率** | 1:1（一个需求一次交互） | 1:N（一次设计，反复受益） |
| **核心产出** | 代码 | 系统/流程/约束 |

---

## 三阶段对比总结

| 维度 | Vibe Coding | Spec Coding | Harness Engineering |
|------|-------------|-------------|---------------------|
| **你在做什么** | 对话 | 规格+验证 | 设计系统 |
| **AI 的角色** | 代码生成器 | 按规格实现的工人 | 被驾驭的持续产出引擎 |
| **你的角色** | 提问者 | 规格设计者+验收者 | 系统架构师 |
| **失败时** | "AI 不行" | "规格没写清楚" | "系统设计有漏洞" |
| **可复制性** | 不可复制（全靠 prompt） | 部分可复制（有测试） | 高度可复制（有完整系统） |
| **团队价值** | 个人提效 | 团队可协作 | 团队能力沉淀 |
| **典型产物** | 代码片段 | 代码+测试 | Skill/Memory/Workflow/Guard Rail |

## 实际例子：同一个需求三种做法

需求：给系统加一个"导出 Excel"功能

### Vibe Coding 做法

```
"帮我写一个 Spring Boot 的 Excel 导出接口"
→ AI 生成代码 → 粘贴进去 → 能跑就行
→ 问题：大数据量 OOM？并发导出？权限校验？不知道
```

### Spec Coding 做法

```
1. 写规格：
   - 支持最大 10 万行导出
   - 超过 1 万行用异步任务 + 下载链接
   - 需要 EXPORT 权限
   - 支持自定义列选择

2. 写测试：
   test_export_small_data_sync()
   test_export_large_data_async()
   test_export_without_permission_403()
   test_export_custom_columns()

3. 让 AI 实现 → 测试全过 → 完成
```

### Harness Engineering 做法

```
1. 设计通用的"功能开发 Skill"（一次设计，反复使用）：
   - 先 brainstorming → 确认方案
   - 再 writing-plans → 拆解任务
   - TDD 开发 → 测试先行
   - verification → 自动验证

2. 设计 Memory：
   - 记住"项目中导出功能统一用 EasyExcel"
   - 记住"大数据量必须走异步"
   - 记住"所有接口都要权限注解"

3. 以后任何人说"加个导出功能"：
   → Agent 自动按 Skill 流程走
   → 自动从 Memory 中获取项目约束
   → 产出质量一致，不依赖某个人的 prompt 能力
```

## 升级路径

```
Level 1 → Level 2 (Vibe → Spec):
  关键转变: 从"问 AI"到"约束 AI"
  具体做法: 学会先写测试/规格，再让 AI 实现
  标志: 你不再说"看起来差不多"，而是说"测试过了"

Level 2 → Level 3 (Spec → Harness):
  关键转变: 从"约束单次任务"到"设计持续系统"
  具体做法: 把你的工作模式编码成 Skill/Memory/Workflow
  标志: 你不再亲自做每件事，而是设计系统让 AI 自主完成
        新人来了不需要你 pair，AI 按你设计的系统自动引导
```
