# G4 集成期 TODO

> **背景**：G4 worktree 实施时 G2 未合入 main，按 plan 的 local-numbering 策略沿用了"假设 G2 不存在"的编号。集成顺序按 plan 仍是 G1 → G2 → G3 → G4 → G5。G4 rebase 到 main（此时 G1+G2+G3 已合）时需要做以下手工调整。

## 1. exit-check.sh 编号统一

**情况**：G4 worktree 内用 `[N/8]`（共 8 个检查，plans 在 `[8/8]`）。G2 合入后会在 main 上加一个 `[8/?]` 沉淀审计（来自 Task 8），并把前 7 项改为 `[N/8]`。

**rebase 时冲突**：`exit-check.sh` 文件末尾两个 `[8/8]` 块（G2 的沉淀审计 + G4 的 plans 状态）都想占位 `[8]`，会冲突。

**手工解决步骤**（在 G4 rebase 过程中或 rebase 后跑）：

```bash
# 1. 保留 G2 的 [8/?] 沉淀审计块不动
# 2. 把 G4 加的 [8/8] plans 段改为 [9/9]
# 3. 全文 [N/8] → [N/9]
sed -i '' 's|\[\([0-9]*\)/8\]|[\1/9]|g' exit-check.sh

# 4. 把 G2 的沉淀审计段标号 [8/9]（sed 上一步应该已经把 G2 的 8/8 改成 8/9，再检查一下）
grep -nE "/[0-9]\]" exit-check.sh
# 期望输出：[1/9] ... [9/9]，且 plans 在 [9/9]，沉淀审计在 [8/9]
```

## 2. CLAUDE.md 同步改号

**情况**：CLAUDE.md 「Plan 系统」段硬编码了 `[8/8]`，并加了 `<!-- INTEGRATION-NOTE: ... -->` 提示。

**集成时**：

```bash
# 删除 INTEGRATION-NOTE 注释 + 改号
sed -i '' 's|`\[8/8\]` 自动列出未完成 plan。<!-- INTEGRATION-NOTE:.*-->|`[9/9]` 自动列出未完成 plan。|' CLAUDE.md
```

或手改一行，把 `[8/8]` 改 `[9/9]`，删掉 HTML 注释。然后 grep 确认无残留：

```bash
grep -n "INTEGRATION-NOTE" CLAUDE.md   # 期望无输出
grep -n "8/8" CLAUDE.md                # 期望无输出
```

```bash

```
